refactor: remove city field from filtering, keep only state and district

Simplified pincode filtering to use only the two core geographic fields:
state and district. Removed city/region/circle from all search/filter operations.

## Rationale:
- City field is always NULL in the database (not in source data)
- Region and Circle are postal administrative hierarchies, not user-facing
- State + District provides sufficient geographic filtering
- Simplifies API, reduces Redis memory, improves maintainability

## Changes:

### API Changes:
- `GET /pincodes` - Removed `city` query parameter
- Now accepts only: `state`, `district`, `search`, `limit`, `page`
- Response DTOs unchanged

### Redis Cache Changes:
- Removed city indexes: `city:index:{city}`
- Removed multi-criteria tables: `lookup:state-city:{key}`, `lookup:district-city:{key}`
- Removed city metadata: `cities:meta`
- Removed city count caches: `count:city:{city}`, `count:state-city:{key}`, `count:district-city:{key}`
- Kept state and district indexes (core filtering)

### Code Changes:
- `PincodeQueryDto`: Removed `city` property
- `PincodeService.findPincodes()`: Removed city filtering logic
- `PincodeService.findPincodesFromDatabase()`: Removed city WHERE clause
- `PincodeCacheService.buildSearchIndexes()`: Removed city index creation
- `PincodeCacheService.getPincodesByFiltersWithPagination()`: Removed city filter branches
- `PincodeCacheService.getCountByFilters()`: Removed city count logic
- `PincodeCacheService.getPincodesByCity()`: Deleted method entirely

### Redis Key Reduction:
**Before:** ~67,740 keys
**After:** ~29,000 keys (est.)

Removed:
- ~19,000 city index keys
- ~18,000 state-city lookup keys  
- ~1,000 district-city lookup keys
- City metadata keys

**Memory Savings:** ~40-50 MB

### Database Schema:
- Pincode entity still has `city`, `region`, `circle` columns
- These are stored in Redis HASHes for completeness
- But NOT indexed for searching/filtering
- Can be displayed in responses, just not filterable

## Result:
- Cleaner, simpler API surface
- Reduced Redis memory footprint
- State + District provides sufficient filtering granularity
- Aligns with actual data availability
